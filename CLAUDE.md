# CLAUDE.md

## What is Homura

Rust ML inference framework: emits MLIR per-kernel → compiles via LLVM to native .so → executes via dlopen with Rust-side buffer orchestration. Runs MNIST, ResNet-18, GPT-2.

## Build & Test

```bash
source env-llvm21-dev.sh
cargo build
cargo test
cargo clippy
cargo fmt -- --check
```

Single test: `cargo test test_name`
Integration tests: `cargo test --test gpt2_e2e` (also: `mnist_e2e`, `resnet18_e2e`)
Examples: `cargo run --example onnx_mnist`, `onnx_resnet`
Download models: `scripts/download_gpt2.sh`, `scripts/download_mnist.sh`

## Architecture

```
ONNX model → per-kernel MLIR emission → parallel compilation → .so per kernel → ExecutionPlan (Rust-side buffer routing)
```

### Compilation pipeline (per kernel)

```
GraphBuilder (MLIR emission) → linalg ops
  → transform schedule (tile + outline + vectorize micro-kernel)
  → bufferize → lower to LLVM dialect
  → LLVM IR → native .o → link to .so
```

### Per-kernel compilation (IREE-style)

Each heavy ONNX op (Conv, MatMul, Gemm) becomes its own kernel with its own
MLIR context + module. Lightweight ops (BatchNorm, Relu, Add) are grouped
between heavy ops. All kernels compile in parallel via rayon.

An `ExecutionPlan` (Rust struct) routes buffers between kernels at runtime
instead of a monolithic MLIR `@compute` driver function.

### Key modules

- `graph_builder.rs` — MLIR emission (GraphBuilder, GraphContext), `finalize_to_mlir`, `compile_from_mlir`
- `compiler.rs` — Shared compilation infrastructure: `build_transform_schedule`, `emit_object_files` (parallel LLVM codegen via `LLVMSplitModule`), `link_shared_lib`
- `runtime.rs` — Execution: `CompiledGraph` (dlopen), `Buffer`, `ExecutionPlan` (multi-kernel orchestration)
- `onnx/emitter.rs` — ONNX → per-kernel emission: `partition_nodes`, `assign_buffer_slots`, `emit_and_compile_plan`
- `onnx/mod.rs` — `Model` struct, `Model::run`, `compile_model_emitter`
- `kv_generate.rs` — KV-cache autoregressive generation

### Transform schedule

`build_transform_schedule` in compiler.rs emits transform dialect sequences:
- Match 3D contractions (matmul), 4D contractions (batched matmul), Conv2D NCHW
- Tile with `tile_using_forall` (cache level) + `tile_using_for` (register level) + `pad`
- Outline the forall loop into a temporary function via `transform.loop.outline`
- Vectorize the outlined function with `vectorize_children_and_apply_patterns`
- LLVM inlines the outlined function during O2 codegen

Kernels without tileable ops (BatchNorm, Relu, elementwise) don't match the
schedule — the transform-interpreter is a no-op, they go straight to scalar
lowering with compact LLVM IR.

## Compilation performance

- MNIST (6 kernels): **97ms** total compile
- ResNet-18 (41 kernels): **467ms** total compile, 256ms parallel kernel compilation
- All kernels compile under 170ms individually
- Per-kernel caching: subsequent runs skip compilation entirely

## LLVM/MLIR bugs

We maintain a patched LLVM 21 branch at `/data2/edgar/llvm-project`. When you
suspect an MLIR bug (crash, miscompile, invalid IR generated by a pass), do NOT
work around it silently. Instead:

1. Write a minimal `.mlir` reproducer
2. Identify the relevant MLIR source file and function
3. Report the full details to the user so they can fix it in our LLVM branch

Known bugs fixed in our branch:
- `tile_using_forall` crash in `normalizeForallLoopOp` on certain IR patterns
- Added `LLVMSplitModule` C binding (`llvm-c/SplitModule.h`)

Known bugs being worked on:
- `vectorize_children_and_apply_patterns` generates invalid `vector.gather`
  from `tensor.extract` with scalar data-dependent indices
